# 00 — 项目宪法（Constitution）

> 文档层级：**宪法** — SDD 体系最上层。这里是**不可协商的原则**，所有规格、计划、任务、代码与所有 AI agent 都必须遵守。
> 稳定性：极高。修改宪法需最高级别评审，且应极少发生。
> 与 Spec 的区别：Spec 写「这一期要实现什么需求」（会变）；宪法写「任何时候都不能违反什么」（几乎不变）。

---

## 适用范围

本宪法约束 `tools/ci/` 下的全部 CI 系统设计与实施，以及由其驱动的所有评测流程。任何文档或代码若与本宪法冲突，以本宪法为准，冲突项必须修正。

---

## 原则

### C-1 信息不外泄（最高优先级）
代码托管、CI 调度、任务执行、结果存储**全部在内网完成**，不依赖任何第三方 SaaS。禁止任何会将代码或数据外传到外部服务的设计。任何便利性都不得凌驾于此原则之上。

### C-2 不可信代码必须隔离
AI 生成的代码一律视为恶意代码。执行前必须置于隔离沙箱：默认断网、限制资源、文件访问最小化（不可见参考答案、凭证、其它题目）、用后即销毁。**禁止为让流程通过而移除或削弱隔离。**

### C-3 仿真资源约束不可绕过
仿真软件不支持并行使用。系统必须保证仿真步骤的并发不超过 License 实例数。**禁止为提升吞吐而违反此约束**导致 License 冲突或工具状态污染。并发度是可配置项，但其上限由真实 License 数决定，不得虚高。

### C-4 商业友好许可
所有 CI 核心组件必须采用 OSI 认证的宽松许可（MIT / Apache 2.0 / GPL 等）。禁止引入 BSL / SSPL 等限制商用自托管的 source-available 许可组件。

### C-5 规格是唯一事实源
代码服务于规格，而非相反。禁止只改代码不回改规格（避免“野生功能”）。每个实现必须可追溯到某条规格需求。

### C-6 变更单向流动
变更方向固定：宪法 → 规格 → 计划 → 任务 → 代码。涉及重要取舍时同时记录 ADR。禁止逆向（如直接改任务而不回看规格）。

### C-7 单一事实源，禁止重复
同一事实只在一处定义，其余处引用而非复制。配置能 include 就不复制粘贴。避免“改一处忘三处”的漂移。

### C-8 一致性可机器校验
规格与实现的一致性必须可由机器检查，并作为 CI 闸门。不一致则流水线失败。不依赖人的自觉来保持对齐。

### C-9 轻量迭代，抵制过度规格
按 feature 迭代增量演进，不追求一次写全的重型前期规格（防“瀑布的复仇”）。治理机制按痛点增量引入，不过度治理。

### C-10 遇阻停止，绝不臆造
执行中遇到缺失信息（凭证、地址等）或验证失败，必须停止并上报，绝不猜测、绕过或编造。
