## 已确定

- **目录**：`tools/ana`（全新空目录），遵循 SDD。
- **领域**：AI 模型在**仿真平台**上运行产生的日志。痛点 = 格式多、内容不规范、分析困难。
- **平台**（有限已知，3 个 → 每平台一个解析器）：
  1. 功能仿真
  2. 性能仿真
  3. 带 OS 软件平台的功能仿真（软件流程多、步骤更杂）
- **日志形态**：纯文本 `.txt`（非 JSON 行）→ 解析走文本/正则路线。
- **整体两层结构**：
  - **一级 · 提取层**：配置提取 / 数据提取 / 异常日志提取
  - **二级 · 分析工具**（后置）：配置可读性提升 / 数据比对 / 异常分析
- **第一版范围（用户选 D）**：只做**提取层**，三类提取跨平台跑通；二级分析工具全部后置。

## 设计草图（待 spec 固化）

```
多平台 AI 模型运行日志(.txt)

   ├─ 数据比对         ← data
   └─ 异常分析         ← exceptions
```

## 待澄清（下次从这里继续，一次问一个）

1. **三类提取物含义确认**（上一个待答问题）——我的理解：
   - 配置 = 这次「怎么跑的」（模型/平台/版本/参数/环境/命令行）
   - 数据 = 这次「产出了什么」（功能结果、性能数字：周期/吞吐/时延、基准值）
   - 异常 = 这次「哪里不对」（ERROR/FATAL、assertion 失败、超时、崩溃栈）
   - ❓ 是否正确？哪类要改？
2. **提取产出形态**：分文件（`config.json`/`data.json`/`exceptions.json`） vs 单 JSON 汇总 vs 人可读文本？
3. **输入粒度**：一次「运行」= 一个 `.txt` 还是一堆 `.txt`？
4. **技术形态**：语言（Python？是否零依赖/离线）、命令行工具 vs 可 import 的库？
5. **真实样例**：各平台脱敏 `.txt` 样例十几行——有了才能定解析规则。
